
## Variable에 관하여.
- 해결책 보다는 문제에 관한 용어로 선택하라. (`어떻게` 가 아니라 `무엇`) - 이건 개체지향 프로그래밍의 성격과도 맥이 닿아있다.
    - `개체지향적 사고`, `구현`, `언어(영어)`의 관계
        - 객체.함수_전치사(파라미터) => 주어.동사_목적어_전치사_파라미터
    - inputRecord 보다는 employeeData, bitFlag 보다는 printerReady 처럼 구체적인 무엇에 관한 명칭이 좋다.
- 8자 ~ 20자 사이의 변수가 통계상 디버깅이 쉬워진다는 연구가 있음. 하지만 꼭 지키기 보다는 저 보다 짧거나 길다면 의심해봐라.
- 은율이 라임을 변수명에 일관성을 부여해라. 그 라임이 접두어이던 접미어이던 선택해서 룰을 가져라. 보통은...
    - 한정자(total, sum, average, index, max, min, num, count)의 위치를 앞에 둘지 뒤에 둘지 결정해라.
- 반의어의 사용은 가독성을 높인다.
    - begin/end, first/last, locked/unlocked, min/max, next/previous, old/new, opened/closed, source/target, source/destination, 
- 상태 변수일 경우 flag라는 이름 사용보다 더 나은 선택지가 있는지 찾아보라. flag 자체가 상태의 의미다. 
- 임시 변수도 이름을 붙여라. 모든 변수는 임시 변수다.
- boolean 타입
    - done, error, found, success, ok. 같은 의미로 사용해라.
    - isDoen, isError, isFound, ...같은 변수명/함수병을 사용해라. ( hasXXX(), canXXX() 도 좋다.)
    - 긍정의 의미로 사용해라. notFound == true 같은 구문은 혼돈을 준다.
- 축약어를 피하라. 
- 0과 1은 그대로 사용해도 괜찮다. 0은 첫번쨰 인덱스를 1은 증감식을 나타내는 걸로 이해가 잘 된다.
- 부동소수점의 덧셈 뺄셈에서 차이가 너무 큰 값들을 연산하지 않는다. 
- 부동소수점의 동치 비교를 하지 않는다.  
- 열거형을 적극 사용해라. boolean도 열거형으로 변경하는게 좋다.

## Strong Type에 관하여.

### 도메인 지식의 변수를 기본 타입으로 선언하기 전 고려사항.
- 변수의 의미가 기존에 선언된 강타입(strong type)과 중복되는지 확인하고, 없다면 알려준다.

### 두 개 이상의 기본타입이 두 번 이상 반복 된다면.
- 2-2 법칙: 두 개 이상의 기본탑이 하나의 의미로 묶여서 두 번 이상 반복 사용될 여지가 있다면, 강타입을 고려하라.

### 함수 파라미터가 많다면, 그 파라미터들은 하나의 추상화된 의미로 전달될 가능성이 매우 높음.
- 강타입을 고려해라.

### immutable / mutable 변수로 사용될지 설계하라.

## class/method 설계의 규칙.

### 파일(class_name.py)의 라인수가 400~600줄을 넘어가면 의심해보자.
- 팀원의 한계를 고려하고, 유지 보수성을 고려하자. 
- 넘어가야만 한다면, 
- 내가 짜지 말아야하는(능력이 좋은사람이 짜야하는) class 일 수도 있고, 
- (설계와 검증) 시간이 많이 필요한 class 일 수도 있고,

### class의 멤버 변수의 개수가 7 +- 2 개에서 벗어나면 의심해보자.
- 개수가 적다면 무의미한 쪼개기인지 의심하고
- 메소드가 필요없는 데이터 클래스를 만들어 사용할 수 있는 변수들인지 의심하고
- 개수가 많아지는건 더욱더 의심해서 class의 역할이 많아지는걸 피해라. 
- 개수가 많아지는건 응집성을 낮추고, 의존성을 높이는 결과를 초래.
- 절대적이진 않지만, 단일 책임 원칙을 한 번 더 생각해본다

### 다른 class와의 의존성이 2개 이상일때 의심해보자.
- pan out을 낮춰라.
- 의존성이 단방향으로 흘러가도록 해라.

### 응집성을 고려하자.
- 절대적이진 않지만, 단일 책임 원칙을 한 번 더 생각해본다
- sin 함수는 한가지만 한다. 
- 예상 가능한 결과를 return한다.

### 함수의 파라미터가 4개를 초과할 경우.
- 함수는 본질적으로 비선형적이라고 가정해야한다.
- 그떄, 비선형적 파라미터가 4개를 초과하면 논리적으로 이해가 힘들어진다.

### 함수 파라미터에 default.
- 하나를 초과하는 함수를 작성하지 말아라.
- 사용자는 default 매개 변수가 많아질 수록 2의 n승 만큼 함수 결과를 예상해야한다.
- 2개만 해도 4가지 함수의 작동방식을 생각하면서 호출을 고려한다. 한 줄 짜려고 너무 많은 리소스를 낭비한다.
- 모두가 예상할 수 있는 default 값을 사용해라.
- index = -1 은 모두 예상 가능. 하지만 bool은 혼돈을 초래 할 수 있으므로 피한다.

### 함수 명은 동사로 시작하라.
- 객체지향적 사고, 구현, 언어(영어)의 관계
- 객체.함수_전치사(파라미터) => 주어.동사_목적어_전치사_파라미터

### 함수 인자로 전달되는 변수 전달/복사 비용을 고려해라.
- 변수가 함수 인자로 전달될때, 복사가 기본적으로 일어나는 경우라면 참조가 되도록 고려해라. (8 byte 기준)

### 자주사용되는 디자인 패턴은 이해하는게 필수다.

## 에러에 관하여.

### exception throw(raise)
- 책임의 문제다.
- 의도적으로 프로세스를 죽일때 사용해라.
- 극악의 확률로 발생할 법한 케이스의 경우에 사용해라.
- 관리가 가능한 코드에서는 사용하지 말자. 외부로 배포되는 패키지일 경우에만 사용해라.
- 거의 사용하지 말라는 뜻이다. 프로세스를 죽일 목적으로 사용하고, 복구해라.
- 왜냐면, 사용하는 함수 안에서 throw를 어디서 하는지 알기 힘들다.

### assert
- 절대 동작 가정 보장을 위한 디버깅 문제다.
- 사용해라. 자주 사용해라.
- 가정에 대한 조건을 넣어라. 이 함수의 동작에 관한 가정을 넣어라.
- assert의 조건과 메시지를 함께 표시해라.
- assert를 사용하고 또 다시 if 문으로 처리하지 말아라. 의미없다. 

### error
- assert의 조건과 구분해라.
- 동작 가정 조건과 동작 실패를 구분해라. 
- 있을 법한 케이스에 대한 error를 정의하고 처리해라.

### Result 패턴을 적용해라.
- 언어가 지원한다면 Result[T, E]를 반환해서, 사용하는 곳에서 처리가 가능하게 해라.
- optional, result, ....

### exception < assert < error(Result) 순으로 사용

## 문서화/주석의 관하여.

### 기본적으로 하지마라.
- 안할 수 있다면 하지마라. `해라`가 기조가 되는 순간, 늪에 빠진다.
- 꼭 필요하고 시간이 지나도 굳건히 밑바탕이 되는 아키텍처 문서, 핵심이 되는 클래스들의 이해를 돕는 문서, 기술 문서 등을 제외하면 유지/보수/관리가 사실상 대부분의 조직에서는 불가능하다. 지속적인 관리의 리소스가 상당하다. 
- 코드 관련된 문서(클래스 관련)는 디테일 할 필요 없다. 전체적인 그림을 보여주면 된다. 코드가 곧 문서가 되게 만들면 된다. 
- 주석에는 목적과 이유를 설명해라. 코드를 설명하지 말아라. 
- 당연하겠지만, 공개 혹은 판매가 되는 코드라면 문서화 해라.

## 테스트에 관하여.

### 테스트 할 수 있는 리소스가 있다면 최대한 해라.
- 하지만, 개발 진행 속도와 트레이드 오프다.

### 테스트로 잡을 수 있는 버그는 대부분 commit되기 전에 고쳐진다.
- 진짜 버그는 스펙 오해로 인한 버그다. (커뮤니케이션)
- 고로 스펙 오해로 생긴 코드에 대한 테스트 코드는 문제 없을것이고, 결과적으로 퀄리티 상승 목적의 의미는 크게 없다.
- 결국 스펙에 대한 옳바른 이해를 위한 커뮤니케이션과 QC/QA다. (결국 비용이고 개발 진행 속도와 트레이드 오프관계에 있다.)

## TMatrix3(Transform Matrix) 변수/함수 명 규칙
> a 기준좌표에서 본 b의 위치/방향, Craig의 표기 방식을 따른다.

``` python

a_T_b: TMatrix3 = get_a_T_b()
a_T_c: TMatrix3 = a_T_b @ b_T_c

b_P_c: Vector3 = get_b_P_c()
a_P_c: Vector3 = a_T_b * b_P_c


```
